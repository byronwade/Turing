# Build Readiness Operating Board

Status: operating handoff board; no status promotion
Owner: program, architecture, documentation, security, release operations, quality, and subsystem owners
Last updated: 2026-07-19

## Purpose

This board tells a maintainer or agent where the project stands before expanding implementation. It connects the pre-build readiness registry, work packages, research questions, evidence reports, and validation commands without creating a second source of truth.

The machine source of truth remains [`pre-build-readiness.json`](../blueprint-v1/machine/pre-build-readiness.json). The first human continuation stop is the [Build Readiness Start Guide](21-build-readiness-start-guide.md), then this one-screen [Build Readiness Progress Snapshot](22-build-readiness-progress-snapshot.md), then the [Build Continuation Readiness Pack](20-build-continuation-readiness-pack.md), and then this operating board. The authoritative executable backlog remains [`backlog.json`](../blueprint-v1/machine/backlog.json) and [Blueprint 19](../blueprint-v1/19-initial-backlog.md). The checked [`PB-020` Implementation Kickoff Review Inventory](../research/implementation-kickoff-review-inventory-2026-07.md) summarizes unresolved-lane first next actions, owner-only decisions, prohibited claims, and release-authority boundaries without promoting readiness. The checked [Build Readiness Dependency Graph](../research/build-readiness-dependency-graph-inventory-2026-07.md) keeps task dependencies and cross-lane sequencing machine-checkable. The checked [Documentation Readiness Completion Audit](../research/documentation-readiness-completion-audit-2026-07.md) and no-claim [build-readiness closure-review template](machine/build-readiness-closure-reviews/no-claim-build-readiness-closure-template.json) keep the documentation-completion claim bounded to contained M0 continuation and reject all-information-ready-for-building, broad M1, Chrome-class, production, release, performance, compatibility, security, and accessibility claims. The checked [Contained M0 Start State Inventory](../research/contained-m0-start-state-inventory-2026-07.md) and machine [`contained-m0-start-state.json`](machine/contained-m0-start-state.json) make session-start routing explicit without approving tasks. The checked [Build Information Readiness Ledger](../research/build-information-readiness-ledger-2026-07.md) and machine [`build-information-readiness-ledger.json`](machine/build-information-readiness-ledger.json) collect the missing information required before broad building or all-information-ready-for-building claims. The proposed [`TASK-*` queue](17-build-readiness-task-queue.md) turns the next blockers into task-shaped handoff records but does not approve execution. This board explains the next operating sequence.

The checked [Chrome-Class Capability Traceability Map](../research/chrome-class-capability-traceability-map-2026-07.md) routes Chrome-class capability domains to owning requirements, blockers, next proof, and prohibited claims. It is traceability only, not Chrome-class evidence, capability evidence, or readiness promotion. The no-claim [Build-Readiness Closure and Owner-Decision Preparation](../research/build-readiness-closure-and-owner-decision-preparation-2026-07.md) defines the evidence order and decision-record handoff required before the closure template can be replaced by a real `PB-020` review.

## Current gate posture

Turing is ready for contained M0 implementation tasks only. It is not ready for broad M1 expansion, developer preview, beta, stable release, or a Chrome-class claim.

Current evidence supports:

- documentation, research, and repository validation work;
- bounded M0 source tasks in the root Cargo workspace;
- typed identity, IPC, kernel-capability, UI-model, build-info, shell-laboratory, and repository-tool work;
- sandbox, benchmark, UI, profile/session, and source-strategy prototypes that do not hard-code unresolved product interfaces.

Current evidence does not support:

- deriving release code from Servo or any other browser engine before `ADR-0009`;
- selecting a UI toolkit before equivalent reference shells and license/provenance review;
- claiming compatibility, security, low memory, speed, accessibility, daily-driver suitability, or production safety;
- running multiple subsystems ahead of unresolved process, IPC, source, sandbox, benchmark, and ownership gates.

## Contained task intake rule

Before creating or executing a contained M0 task, record:

- stable task ID, owner, reviewer, and rollback owner;
- linked `PB-*`, `WP-*`, `REQ-*`, `RISK-*`, `ADR-*`, and `RQ-*` records;
- exact files and crates in scope;
- explicit non-scope and unsupported behavior;
- required negative tests and failure modes;
- dependency, unsafe, native, generated-code, provenance, and license impact;
- expected documentation updates;
- validation commands and evidence bundle location.

If the task crosses trust boundaries, browser UI state, source strategy, process capability, IPC, storage, update, agent authority, or release scope, it needs a reviewed design record before source work.

Before executing proposed `TASK-000001` through `TASK-000010`, also verify [`contained-m0-start-state.json`](machine/contained-m0-start-state.json): the queue remains owner-approval-required and no proposed row is executable until a reviewed immutable manifest replaces the no-claim approval template for that task.

## Drift controls

Every continuation must keep these records synchronized:

- [`docs/README.md`](../README.md) for navigation;
- [`docs/repository-map.md`](../repository-map.md) for structure;
- [research log](../research-log.md) for material research or governance changes;
- owning Blueprint chapter and detailed book;
- machine registry for any readiness, work-package, requirement, risk, owner, review, or gate meaning change;
- validation tools when a new invariant becomes mandatory.

Do not update this board as a substitute for updating the canonical source. If a board row disagrees with the machine registry, the machine registry wins and this board is stale.

## Validation entry points

Before handoff, run one aggregate validation wrapper:

```bash
sh tools/check.sh
```

```powershell
.\tools\check.ps1
```

Both wrappers delegate to `cargo run --locked -p xtask -- check` and keep Cargo build output outside the repository when `CARGO_TARGET_DIR` is unset. The full direct command list is maintained in the [Documentation Readiness Evidence Matrix](18-documentation-readiness-evidence-matrix.md#validation-commands). A passing check proves only the contained M0 repository validation scope.

## Stop conditions

Stop and open a design or owner review before:

- importing browser-engine, JavaScript-runtime, WebView, Electron, CEF, or system-webview code into a release path;
- adding external runtime dependencies to M0 crates;
- using unsafe or native code without ledger entries and review;
- selecting a UI toolkit, process topology, IPC wire encoding, storage format, update trust model, benchmark claim, or stable platform;
- weakening validation, sandbox, site isolation, certificate, update, or permission policy to make a demo work;
- claiming Chrome-class, faster, lower memory, secure, compatible, accessible, production-ready, or daily-driver status without the evidence demanded by [Blueprint 20](../blueprint-v1/20-definition-of-done.md).

## Handoff checklist

A person continuing the project should be able to answer:

1. What gate does this work advance?
2. Which source of truth owns the claim?
3. What evidence exists now?
4. What evidence is missing?
5. What must not be claimed yet?
6. What commands prove the repository still validates?

If any answer is unclear, update the owning document or registry before expanding implementation.
